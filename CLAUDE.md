# Team System — Operating Manual

## Team Roster

| Agent | Role | Model | When |
|-------|------|-------|------|
| **product-owner** | Scope, approve plans, completion | Opus | Planning, review, summary |
| **software-engineer** | Plans, implements, tests, fixes | Opus | Planning, implementation |
| **triage** | Fast pre-filter: research, parse data, summarize | Haiku | Before every review agent |
| **code-health** | Refactoring, simplification, tech debt, deps | Sonnet | Post-feature, on-demand |
| **devsecops** | Security scan, infra, compliance | Sonnet | Plan review, post-implementation |
| **qa-engineer** | Validates ACs, edge cases, coverage | Sonnet | After code review |
| **code-reviewer** | Code quality, best practices | Sonnet | After implementation |
| **ux-designer** | UI/UX specs, accessibility | Sonnet | During UI implementation |
| **mlops** | ML pipelines, training, serving, experiments | Sonnet | Only for ML projects |

All agents are **stack-aware** — they read `.claude/stack.md` and the relevant
language profiles from `.claude/stacks/` to adapt their reviews, testing, and
security checks to the project's actual tech stack.

## Feature Directory Structure

Each feature has its own directory under `docs/features/`:

```
docs/features/
├── .active                          ← contains name of active feature
├── patient-intake/                  ← /scope patient-intake ...
│   ├── sow.md                       ← Requirements (from PO)
│   ├── technical-plan.md            ← Plan (from SE)
│   ├── devsecops-plan-review.md     ← Infra/security review (from DevSecOps)
│   ├── plan-approved.md             ← Approval gate (from PO review)
│   ├── code-review.md               ← Review (from Code Reviewer)
│   ├── security-scan.md             ← Vulnerability scan (from DevSecOps)
│   ├── qa-report.md                 ← Validation (from QA)
│   ├── refactor-report.md           ← Post-feature cleanup (from Code Health)
│   ├── pr-description.md            ← PR body (generated by /ship)
│   ├── project-summary.md           ← Completion (from PO)
│   └── scans/                       ← Automated scanner output
│       ├── scan-summary.md          ← What ran, what found issues
│       ├── semgrep.json             ← SAST findings
│       ├── gitleaks.json            ← Secrets detection
│       └── ...                      ← Language-specific reports
├── auth-system/                     ← /scope auth-system ...
│   └── sow.md                       ← (still in planning)
└── reporting/                       ← /scope reporting ...
    ├── sow.md
    ├── technical-plan.md
    └── plan-approved.md             ← (ready to build)
```

## Implementation Gate

The gate is **optional** — only activates when an active feature has a SOW:
- **No active feature or no SOW** → gate inactive → code freely
- **SOW exists, no plan-approved.md** → gate blocks implementation code
- **plan-approved.md exists** → gate open → code freely
- **Always allowed**: docs/, tests, configs, .claude/, CLAUDE.md, package files

Bypass: `/approve-plan`, `/switch` to approved feature, or `/fresh --clear`

## The Pipeline

```
/scope [description]                           ← INTERACTIVE (you + PO)
│
├─ Auto-generate feature name + git branch
├─ Triage (Haiku): research best practices
├─ PO (Opus): Discovery → questions → scope → SOW
│     ↕ You discuss scope, edge cases, best practices with PO
├─ SE (Opus): Technical plan
├─ DevSecOps (Sonnet) + MLOps (if ML): Plan review
├─ Triage (Haiku): Plan structure check
├─ PO (Opus): Reviews plan → approve or reject
│     ↕ You review the plan
│
╔═══════════════════════════════════════════════════════╗
║  /build-phase all                 ← AUTOPILOT MODE   ║
╠═══════════════════════════════════════════════════════╣
║  Build: SE implements each phase + tests + commit     ║
║  Scan:  Scanners → Triage (Haiku) parses results      ║
║  Review: Code Reviewer + DevSecOps (parallel)         ║
║  QA: validates ACs → SE fixes if needed (2 rounds)   ║
║  2nd Pass: focused review on fix commits only         ║
║  Cleanup: Code Health refactors + simplifies          ║
║  → Presents results, you review and test              ║
╚═══════════════════════════════════════════════════════╝

/ship          ← After you've reviewed: generates PR description, creates draft PR
/review        ← Standalone: run review pipeline on any branch (no plan needed)
/health        ← On-demand code health checks
```

## Conventions
- All agents read from the active feature directory (docs/features/{name}/)
- Read the SOW before implementing anything
- Follow existing codebase patterns over introducing new ones
- Write tests alongside code, not after
- If deviating from the plan, document WHY

## Hooks Behavior

The hooks in `.claude/settings.json` are designed to coexist with normal Claude Code:

- **No active feature** (`docs/features/.active` doesn't exist): hooks do nothing, Claude Code works normally
- **Active feature, plan approved**: gate is open, code freely
- **Active feature, plan NOT yet approved**: gate blocks implementation files (docs/tests/config always allowed)
- **Any error or missing scripts**: hooks fail-open, never blocks unexpectedly
- **Built-in commands** (`/simplify`, `/batch`, etc.): always work, hooks only chain for team agents
